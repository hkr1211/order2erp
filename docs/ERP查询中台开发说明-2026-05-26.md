## 1. 项目背景

公司现有智邦 ERP 已运行多年，ERP 内部数据较完整，但业务人员、管理层和老板难以从繁杂的 ERP 页面中快速识别风险。当前项目目标是建设一个轻量级“ERP 查询中台”，通过 API 同步 ERP 数据到本地 SQLite，再基于本地数据提供 PMC、订单、物料、生产、派工、财务、报价、异常、报表等业务页面。

项目当前不做 ERP 替代，也不做生产报工回写。车间报工继续使用 ERP。本系统主要解决：

- 老板一眼看到公司交期、缺料、资金、待报价等风险。
- 管理层快速定位瓶颈和责任区域。
- 销售/跟单员看到自己负责订单的待干预事项。
- 财务看到应收应付和逾期风险。
- 后续为 OpenClaw、Hermes、飞书机器人等自然语言入口提供统一数据服务。

## 2. 当前技术栈

- 后端：Node.js 原生 HTTP 服务，ES Module。
- 数据库：SQLite，默认文件 `data/pmc.db`。
- ERP 接入：智邦 ERP OpenAPI + 部分旧版 ASP 接口。
- 前端：服务端渲染 HTML，未引入 Vue/React。
- 测试：Node 自带 `node:test`，通过 `npm run check` 执行语法检查和单元测试。
- 运行方式：`npm start`。

核心目录：

```text

  server.js             页面路由、API 路由、HTML 渲染
  erpClient.js          智邦 ERP API 客户端和业务视图转换
```

## 7. ERP 保护机制

由于 ERP 曾出现 503、卡死、`fetch failed`，项目已加入多层保护：

- `ERP_PROTECTION_MODE`：保护模式下禁止启动时自动压 ERP。
- ERP 请求队列：控制并发和最小请求间隔。
- 熔断机制：连续失败达到阈值后暂停实时请求。
- 同步暂停开关：`/sync-pause?state=on` 可禁止手动同步和历史同步入口。
- 请求日志：`erp_request_logs` 记录路径、状态、耗时和错误。
- 同步失败保留旧数据，不清空 SQLite。

当前经验：

- 财务旧版接口较稳定，可连续分页。
- 库存接口 `InventorySummary` 第 91 页多次失败。
- 库存明细 `InventoryDetails` 第 53 页多次失败。
- 派工计划 `procedure_plans` 第 56 页多次失败。
- 仓库过滤参数 `cks` 对库存接口未能可靠限制仓库，传仓库名或仓库 ID 都可能返回其它仓库数据。

## 11. 已知问题与技术债

### 11.1 架构层面

- 当前 `server.js` 体量较大，页面渲染、API、业务逻辑混在一起，后续建议拆分为路由、服务、视图模板。
- 前端是服务端拼 HTML，适合快速验证，但长期可维护性有限。
- 没有认证和正式权限系统，当前为内网免登录版。
- SQLite 单文件适合当前阶段，若多人高频访问和历史数据继续扩大，需评估 MySQL/PostgreSQL。

### 11.2 数据层面

- 合同明细没有独立本地表，订单详情仍有部分依赖实时或临时数据。
- BOM 展开、采购在途、质检、物流、设备状态缺失。
- 物料齐套目前按销售订单产品库存粗算，未做到工单+BOM+在途采购口径。
- 废料库无法定向同步，需智邦确认库存接口正确过滤字段。
- 部分 ERP 接口按页返回时会在特定页失败，需要确认是数据异常、分页参数问题还是 ERP 接口限制。

### 11.3 业务闭环层面

- 干预按钮已有雏形，但“处理动作 -> 责任人 -> 截止时间 -> 结果复盘”还不完整。
- 角色工作台已有入口，但仍需要根据真实组织架构配置负责人和权限。
- 报表更多是当前快照汇总，趋势分析和月度对比还不够。

## 12. 建议架构师重点评估的问题

1. 当前 Node.js + SQLite 方案是否适合继续作为 V1 内网系统，还是应尽快切换到 Spring Boot/MySQL。
2. ERP 同步应采用“按页轮询”还是建设更明确的增量同步机制。
3. 智邦 ERP 库存接口失败页的根因如何定位，是否需要数据库直连或 ERP 厂商提供专用视图。
4. 是否需要引入任务调度器，例如 cron/XXL-JOB，替代当前手动历史同步。
5. 权限系统应如何设计：内网免登录、飞书单点、账号密码，还是 ERP 用户映射。
6. PMC 风险模型应如何从“规则提醒”升级到“闭环干预系统”。
7. 若接入飞书/自然语言问答，哪些数据必须脱敏，哪些角色能看到哪些范围。
8. SQLite 历史数据保留策略、备份策略、迁移策略如何设计。

## 14. 当前结论

当前系统已经从“ERP API 测试工具”发展为一个可运行的内网 PMC 查询中台原型。它已经具备页面入口、SQLite 缓存、ERP 限速保护、历史同步、数据覆盖率、PMC 驾驶舱、订单/物料/生产/派工/财务/异常/报表等核心模块。

但要进入稳定生产使用，还需要重点补强三件事：

- 数据完整性：库存、派工、合同明细、采购在途、BOM。
- 闭环能力：风险处理记录、责任人、升级机制、复盘。
- 架构治理：模块拆分、权限、任务调度、数据库升级评估。
